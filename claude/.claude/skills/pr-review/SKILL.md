# PR Review スキル

PRレビューを段階的に進める。diffだけ見て雰囲気でコメントするのではなく、ソースコードまで読み込んで根拠のある指摘を出す。

## 使用例

- `/pr-review 1234` → PR #1234をレビュー
- `/pr-review` → 現在のブランチのPRをレビュー

## ワークフロー

### Phase 1: PR情報の取得

PR IDの有無で取得方法を切り替える。

```
# PR IDが指定された場合
gh pr view <PR_ID>
gh pr diff <PR_ID>
gh pr checks <PR_ID>

# PR IDが省略された場合（現在のブランチ）
gh pr view
gh pr diff
gh pr checks
```

取得した情報から以下を把握する:
- PRのタイトル・説明文
- 変更ファイル一覧と行数
- CIの状態

**gh コマンドが失敗した場合:**
- `gh auth status` で認証状態を確認する
- PRが見つからない場合は、ブランチ名やリモートの状態を `git branch -vv` で確認する
- それでも解決しない場合はユーザーにPR URLを直接聞く

### Phase 2: PRタイプの判定

PRの内容から以下のタイプを判定する。タイプごとにレビューの力点が変わる。

| タイプ | 判定基準 | 重点レビュー観点 |
|---|---|---|
| 新機能 | 新規ファイル追加が多い、feature系ブランチ | 設計の妥当性、テストカバレッジ、既存コードとの整合性 |
| バグ修正 | fix/bugfix系ブランチ、小規模な変更 | 根本原因の対処か、副作用の有無、再発防止のテスト |
| リファクタリング | refactor系ブランチ、振る舞い変更なし | 振る舞いが変わっていないこと、テストが維持されていること |
| 依存関係更新 | lock/gemfileの変更、dependabot | 破壊的変更の有無、セキュリティアドバイザリ |
| 設定変更 | CI/CD、インフラ、環境変数 | 本番環境への影響、ロールバック可能性 |

判定が難しい場合は複数タイプの観点を併用する。

### Phase 3: コード調査

ここが最も重要なフェーズ。diffだけでなく、変更ファイルのソースコードを実際に読む。diffは「何が変わったか」しか見えない。呼び出し元や既存のテストを読まないと「その変更が正しいか」は判断できない。

**変更規模の判定**

- 小規模（変更ファイル5個以下、差分100行以下）: 直接読む
- 中規模（変更ファイル6-15個、差分100-500行）: ファイルを領域ごとにグループ化して順に読む
- 大規模（変更ファイル16個以上、差分500行超）: 並列エージェントで調査

**大規模PRの並列調査**

変更ファイルを領域（バックエンド/フロントエンド/テスト/設定など）でグループ化し、各グループをExploreエージェントに委譲する。1つのレスポンスで全てのAgentツールを同時に呼び出すことで並列実行される。順番に1つずつ呼ぶとレビュー時間が倍以上かかるため、必ず同時に起動する。

```
Agent({
  description: "バックエンド変更を調査",
  subagent_type: "Explore",
  prompt: "以下のファイルの変更を調査してください。
    diffの内容だけでなく、各ファイルのソースコード全体を読んで、
    変更の文脈・影響範囲・問題点を報告してください。
    変更ファイル: {バックエンド系ファイルのリスト}
    diff内容: {該当部分のdiff}"
})

Agent({
  description: "フロントエンド変更を調査",
  subagent_type: "Explore",
  prompt: "以下のファイルの変更を調査してください。
    diffの内容だけでなく、各ファイルのソースコード全体を読んで、
    変更の文脈・影響範囲・問題点を報告してください。
    変更ファイル: {フロントエンド系ファイルのリスト}
    diff内容: {該当部分のdiff}"
})
```

**調査時に確認すること**

- 変更されたファイルの全体構造（クラス、モジュール、関数の関係）
- 変更箇所を呼び出している側のコード
- 変更箇所と関連するテストファイル
- 同じパターンが他の箇所にもあるか（修正漏れの可能性）

### Phase 4: レビュー実行

Phase 2で判定したタイプと、Phase 3の調査結果を踏まえてレビューする。

**フィードバックラダー**

指摘はすべて以下のレベルに分類する。レベルを明示することで、レビューイが優先度を判断しやすくなる。

1. **blocker**: マージ前に必ず修正が必要。セキュリティ脆弱性、データ損失リスク、本番障害の可能性
2. **critical**: 強く修正を推奨。パフォーマンス問題、設計上の問題、将来の技術的負債
3. **suggestion**: 検討してほしい改善案。可読性向上、より良いパターンの提案
4. **nitpick**: 任意。命名、スタイル、コメントの改善
5. **praise**: 良い実装の認識。学ぶべきパターン、エレガントな解決策

**レビューの原則**

- 根拠なき指摘はしない。「〜のほうが良い」ではなく「〜だから〜のほうが良い」と書く
- 修正案を具体的に示す。問題の指摘だけで終わらない
- PRの目的外の改善は指摘しない。スコープ外のリファクタリング提案は控える
- テストが書かれていない場合、どんなテストが必要かを具体的に示す

### Phase 5: セルフチェック

レビュー結果を別の視点で検証する。レビュアー自身が思い込みで間違えることは珍しくない。ソースコードの読み間違い、PRスコープの誤解、過剰な厳しさなどを自分で潰す。

確認項目:
- 指摘の根拠は正しいか（ソースコードの読み間違いがないか）
- 的外れな指摘がないか（PRの目的を誤解していないか）
- 見落としがないか（セキュリティ、エラーハンドリング、エッジケース）
- blockerレベルの指摘が本当にblockerか（過剰に厳しくないか）
- praiseすべき良い実装を見落としていないか

大規模PRの場合は、セルフチェックもExploreエージェントに委譲する:

```
Agent({
  description: "レビュー結果を検証",
  subagent_type: "Explore",
  prompt: "以下のPRレビュー結果を検証してください。
    PR概要: {PRの概要}
    レビュー結果: {Phase 4の出力}
    確認観点:
    - 指摘の根拠がソースコードと一致しているか
    - 的外れな指摘がないか
    - 見落とされている問題がないか"
})
```

### Phase 6: 最終出力

以下のフォーマットで出力する。

---

## PR Review: #{PR番号} {PRタイトル}

### 概要

{このPRが何をしているか、1-3文で}

PRタイプ: {新機能 / バグ修正 / リファクタリング / 依存関係更新 / 設定変更}
CI: {pass / fail / pending}

### 総合評価: X/10

{スコアの根拠を1文で}

### 指摘事項

#### blocker

- **{ファイル名}:{行番号}** - {指摘内容}
  - 根拠: {なぜ問題なのか}
  - 修正案: {具体的な修正方法}

#### critical

- **{ファイル名}:{行番号}** - {指摘内容}
  - 根拠: {なぜ問題なのか}
  - 修正案: {具体的な修正方法}

#### suggestion

- **{ファイル名}:{行番号}** - {指摘内容}

#### nitpick

- {内容}

#### praise

- **{ファイル名}:{行番号}** - {良い点}

### 影響範囲

- {この変更が影響する他のコンポーネント・機能}

### 人間が確認すべきポイント

- {AIでは判断が難しい業務ロジックの妥当性}
- {ドメイン知識が必要な判断}
- {ユーザー体験への影響}

---

**スコアリング基準**

- 10: 問題なし。そのままマージ可能
- 8-9: 軽微な改善点あり。nitpick/suggestionのみ
- 6-7: 改善推奨の箇所あり。criticalが1-2件
- 4-5: 修正が必要。criticalが複数、またはblockerが1件
- 1-3: 大幅な修正が必要。blockerが複数
