# Code Research スキル

PRレビュー中、特定の変更箇所について「どういう変更か / データがどう加工されるか / 振る舞いが変わっていないか」を短時間で把握する。

`pr-review` の Phase 3（コード調査）をより軽量・単独で回すための道具。本体は単独で完結する。

## 使用例

- `code-research app/services/foo.rb get_rank_up_coupons`
- diff 断片を貼って「このメソッドの振る舞い変わってる?」
- `code-research get_rank_up_coupons`（シンボルだけ、現在のブランチから git grep で特定）

## 入力（柔軟）

以下いずれでも受け付ける:

1. **ファイル + メソッド名**: `app/services/foo.rb get_rank_up_coupons`
2. **diff 断片を貼る**
3. **シンボルだけ**: `get_rank_up_coupons`（現在のブランチから git grep で特定）
4. **組み合わせ**

不明点・曖昧さがあればユーザーに聞き返す。勝手に推測して進めない。

## 調査フロー

### 1. 対象特定

- 入力を解釈し、調査対象のファイル・メソッド・差分範囲を確定する
- シンボルだけの場合は `git grep -n "def <method>"` で候補を列挙
- 複数候補ヒットしたら選択肢を提示してユーザーに選んでもらう

### 2. 旧実装と新実装を並べる

```
# ベースコミットを特定
git merge-base HEAD <base_branch>

# 旧実装を取得
git show <base>:<file>
```

- 新実装は作業ツリーから Read で読む
- 旧と新を手元で対比できる状態を作る

### 3. データフロー追跡（重要）

メソッド入口から出口まで、**どの変数がどう加工されるかを最初から最後まで追う**。

- shape 変換（配列 → ハッシュ、`group_by`、`transform_values`、`map`、`reduce` 等）を明示
- 旧と新の両方で追って対比
- 中間変数の型・構造を1ステップずつ書き出す

### 4. 6観点チェック（該当ありのみ出力）

1. **振る舞い差分**: 返すデータの**範囲/スコープ**変化を含む（例: 現ランク以降 → 全ランク）、判定ロジック、分岐の変化
2. **データshape変換**: `group_by`, `transform_values`, `map`, `reduce` 等の変換経路
3. **payload / 返り値フィールド差分**: ハッシュのキー・値の比較表
4. **呼び出し元**: このメソッドを使っている他の箇所（修正漏れ候補）。`git grep` で特定
5. **テストの有無**: 対象メソッドのspecがあるか、変更に追従しているか、**未追従/不在の区別**
6. **副作用**: DB書き込み、外部API、通知、キャッシュなどの発生箇所の変化

**該当しない観点のセクションは省略する。**

### 5. 確認事項（必須セクション）

- 自動で判断できない項目、人間が見るべきポイントを箇条書きで出す
- 「スコープ拡大は意図通りか」「呼び出し元Xは新しい返り値shapeに対応済みか」など

## 出力形式

````markdown
## 調査対象: <file>:<method>

## データフロー（最初 → 最後）

旧:
<入力 → 加工1 → 加工2 → 出力>

新:
<入力 → 加工1 → 加工2 → 出力>

## 振る舞い差分

- スコープ: <旧: 現ランク以降 / 新: 全ランク> など
- 判定ロジック: ...
- 返り値: ...

## payload差分

| フィールド | 旧 | 新 |
|---|---|---|
| id | coupon.id | coupon&.id |
| members_card_rank_id | （なし） | rank.id |
| ... | ... | ... |

## 呼び出し元

- path/to/file.rb:123
- path/to/other.rb:45

## テスト

- spec/services/foo_spec.rb - 追従済み / 未追従 / 不在

## 副作用

（該当なければセクションごと省略）

## 確認事項

- [ ] ...
- [ ] ...
````

## 前提と非対象

- **Rails / Ruby 前提**で作る（最初はここにフォーカス）
- PRレビュー全体を回すタスクは `pr-review` スキルを使う
- バッチ調査（複数メソッド一括）は今はやらない（YAGNI）
