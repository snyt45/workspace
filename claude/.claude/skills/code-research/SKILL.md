# Code Research スキル

PRレビュー中、特定の変更箇所について「どういう変更か / データがどう加工されるか / 振る舞いが変わっていないか」を短時間で把握する。

`pr-review` の Phase 3（コード調査）をより軽量・単独で回すための道具。本体は単独で完結する。

## 使用例

- `code-research app/services/foo.rb get_rank_up_coupons`
- diff 断片を貼って「このメソッドの振る舞い変わってる?」
- `code-research get_rank_up_coupons`（シンボルだけ、現在のブランチから git grep で特定）

## 入力（柔軟）

以下いずれでも受け付ける:

1. **ファイル + メソッド名**: `app/services/foo.rb get_rank_up_coupons`
2. **diff 断片を貼る**
3. **シンボルだけ**: `get_rank_up_coupons`（現在のブランチから git grep で特定）
4. **組み合わせ**

不明点・曖昧さがあればユーザーに聞き返す。勝手に推測して進めない。

## 調査フロー

### 1. 対象特定

- 入力を解釈し、調査対象のファイル・メソッド・差分範囲を確定する
- シンボルだけの場合は `git grep -n "def <method>"` で候補を列挙
- 複数候補ヒットしたら選択肢を提示してユーザーに選んでもらう

### 2. 旧実装と新実装を並べる

```
# ベースコミットを特定
git merge-base HEAD <base_branch>

# 旧実装を取得
git show <base>:<file>
```

- 新実装は作業ツリーから Read で読む
- 旧と新を手元で対比できる状態を作る

### 3. データフロー追跡（重要）

メソッド入口から出口まで、**どの変数がどう加工されるかを最初から最後まで追う**。

**書き方のルール**:
- 生コードをそのまま貼らない。**日本語 + 短い擬似コード**で抽象化する
- 1ステップ1行。`ステップ名: 入力 → 出力（shape変化の説明）` の形式
- 中間変数の型・構造を明示（例: `配列[Coupon]` / `Hash{cc_id => Coupon}`）
- 旧と新を**同じステップ粒度**で並べて対比しやすくする

悪い例（読めない）:
```
user.coupons.joins(:coupon_content).not_expired.where(status: "active").where(coupon_content_id: members_card_rank.coupon_content_ids)...
```

良い例:
```
1. 対象クーポン取得: user.coupons を active かつ未期限切れで絞り込み → 配列[Coupon]
2. ランク別変換: 現ランクと next_ranks ごとに payload 化 → 配列[Hash]
3. フラット化 → 配列[Hash] を返す
```

### 4. 6観点チェック（該当ありのみ出力）

各項目に**重大度マーク**を付けて出力する:
- 🔴 **重大**: 既存の呼び出し元/クライアントが壊れる可能性。必ず確認が要る
- 🟡 **要確認**: 意図通りかユーザー判断が必要。壊れないが挙動が変わる
- 🟢 **軽微**: 等価リファクタ、命名変更など

**観点**:
1. **振る舞い差分**: 返すデータの**範囲/スコープ**変化（例: 現ランク以降 → 全ランク）、判定ロジック、分岐の変化
2. **データshape変換**: `group_by`, `transform_values`, `map`, `reduce` 等の変換経路
3. **payload / 返り値フィールド差分**: ハッシュのキー・値の比較表。追加・削除・型変化にマーク
4. **呼び出し元**: このメソッドを使っている他の箇所（修正漏れ候補）。`git grep` で特定
5. **テストの有無**: 対象メソッドのspecがあるか、変更に追従しているか、**未追従/不在の区別**
6. **副作用**: DB書き込み、外部API、通知、キャッシュなどの発生箇所の変化

**該当しない観点のセクションは省略する。**

### 5. 冒頭サマリ（必須）

6観点の調査が終わったら、**最初に `## 要点` セクションを書く**。
- 🔴重大な差分を2〜4行で箇条書き
- 🔴が無ければ「重大な差分なし」と明記した上で🟡を並べる
- レビュアーが冒頭3行だけ読めば「どこを見るべきか」が分かる状態を作る

### 6. 確認事項（必須セクション）

- 自動で判断できない項目、人間が見るべきポイントを箇条書きで出す
- **各項目に優先度マーク**を付ける: `[重要]` / `[通常]`
- `[重要]` を上に並べる。スコープ拡大・payload破壊・テスト不在などは基本 `[重要]`

## 出力形式

````markdown
## 調査対象: <file>:<method>

## 要点

- 🔴 <最も注意すべき変更を1行で>
- 🔴 <次に注意すべき変更>
- 🟡 <意図確認が必要な点>

## データフロー（最初 → 最後）

旧:
1. <ステップ名>: <入力> → <出力（shape）>
2. <ステップ名>: <入力> → <出力（shape）>
3. ...

新:
1. <ステップ名>: <入力> → <出力（shape）>
2. <ステップ名>: <入力> → <出力（shape）>
3. ...

## 振る舞い差分

- 🔴 スコープ: 旧=現ランク以降 / 新=全オープンランク（過去の達成済みも含む）
- 🔴 絞り込み: 旧=active+未期限のみ / 新=全status（使用済・期限切れも返る）
- 🟡 判定ロジック: ...
- 🟢 命名変更のみ: ...

## payload差分

| フィールド | 旧 | 新 | 重大度 |
|---|---|---|---|
| id | coupon.id | coupon&.id（nil可） | 🔴 |
| members_card_rank_id | （なし） | rank.id | 🟡 新規 |
| status | （なし） | coupon&.status | 🟡 新規 |
| disabled | !is_current_rank | rank.id != members_card_rank_id | 🟢 等価 |

## 呼び出し元

- path/to/file.rb:123
- path/to/other.rb:45

## テスト

- spec/services/foo_spec.rb - 追従済み / 未追従 / 不在

## 副作用

（該当なければセクションごと省略）

## 確認事項

- [ ] [重要] スコープ拡大（全オープンランクを返す）は意図通りか
- [ ] [重要] payload の id が nil になりうる挙動にフロントが対応しているか
- [ ] [重要] spec 不在のまま進めて良いか
- [ ] [通常] 新規フィールド members_card_rank_id のクライアント側対応状況
````

## 前提と非対象

- **Rails / Ruby 前提**で作る（最初はここにフォーカス）
- PRレビュー全体を回すタスクは `pr-review` スキルを使う
- バッチ調査（複数メソッド一括）は今はやらない（YAGNI）
