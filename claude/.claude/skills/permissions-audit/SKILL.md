# permissions-audit

Claude Codeの3つのsettings.jsonを読み込み、permissions設定とプラグイン状態を棚卸しする。

## 対象ファイル

以下の3ファイルを順に読み込む。存在しないファイルはスキップする。

1. `~/.claude/settings.json` - グローバル設定（全プロジェクト共通）
2. `.claude/settings.json` - プロジェクト設定（チーム共有、gitにコミットされる）
3. `.claude/settings.local.json` - ローカル設定（個人用、gitignore対象）

## 実行手順

### Step 1: ファイル読み込みと一覧化

3ファイルをReadツールで読み込む。各ファイルからpermissionsセクションとenabledPluginsセクションを抽出し、以下の形式で一覧表示する。

```
## 現在の設定一覧

### グローバル (~/.claude/settings.json)
allow (N件): ...
deny (N件): ...
ask (N件): ...
enabledPlugins: ...

### プロジェクト (.claude/settings.json)
（未設定 or 内容）

### ローカル (.claude/settings.local.json)
（未設定 or 内容）
```

件数とルール名を列挙する。allowはカテゴリごとにグルーピングすると見やすい（ツール系、ファイル操作、git、gh、パッケージマネージャ、シェルコマンド等）。

### Step 2: 診断

以下の観点で問題を検出する。問題がなければ「問題なし」と明示する。

#### 2-1. スコープの妥当性

グローバルに置くべきもの（汎用的な読み取りコマンド、git基本操作など）がプロジェクトやローカルにある、またはその逆を検出する。

判断基準:
- グローバル向き: どのプロジェクトでも使う汎用コマンド（ls, cat, git status等）
- プロジェクト向き: 特定の技術スタック固有のコマンド（rails, cargo, docker等）
- ローカル向き: 一時的な許可、個人の開発ツール固有のもの

ローカルに溜まったルールは「意図的に残しているか、掃除し忘れか」を確認する。

#### 2-2. 重複・矛盾の検出

3つの観点で確認する:
- 同じルールが複数ファイルに存在する（スコープ間の重複）
- allowとdenyで同じパターンが競合する（矛盾）
- 広いワイルドカードが狭い個別ルールを包含している（冗長）
- 専用ツール（Glob, Read, Grep）とBashコマンド版（find, cat, grep, rg等）の機能重複。これらは残しても実害はないが、整理すると見通しが良くなる。判断はユーザーに委ねる

#### 2-3. 過剰な許可の警告

以下のパターンを検出する。警告レベル（高/中/低）を必ず明示する。

高:
- `Bash(*)` のような全許可
- `Write` が無制限（パス指定なし）でallowにある
- `Edit` が無制限でallowにある

中:
- `Bash(npx:*)` がallowにある（外部パッケージのダウンロード+実行が可能）
- `Bash(node:*)` がallowにある（任意コード実行が可能）
- `Bash(curl:*)` や `Bash(wget:*)` がallowにある
- `Bash(chmod:*)` がallowにある
- `Bash(rm:*)` がallowにある
- パッケージインストール系がallowにある

低:
- 整理すると見通しが良くなる程度のもの

方針の不整合も検出する。同じ種類の操作なのに片方がallow、もう片方がaskになっているケースに注目する。

典型例:
- `WebFetch` がallowなのに `Bash(curl:*)` / `Bash(wget:*)` がask（外部通信の方針不一致）
- `Bash(npx:*)` がallowなのに `Bash(npm install:*)` がask（パッケージ取得の方針不一致）
- `Write` が無制限なのに `Edit` にパス制限がある（書き込み範囲の不一致）

#### 2-4. 不足の指摘

よくある設定で抜けているものがあれば指摘する。ただし押し付けない。
- denyに .env 系が入っていない
- askに破壊的git操作（push, reset, checkout, rebase）が入っていない
- denyに入れるべき危険なコマンド（sudo, dd, shutdown等）が未登録

#### 2-5. enabledPluginsの確認

有効なプラグインを一覧化し、以下を確認する:
- 使っていないプラグインが有効のまま残っていないか
- 一時的に有効にしたプラグインが戻し忘れになっていないか（MEMORYやCLAUDE.mdに運用ルールがあればそれと照合する）
- 現在のプロジェクトに不要なプラグインが有効になっていないか

### Step 3: 整理提案

検出した問題に基づいて、具体的な修正案を優先度付きで提示する。

提案の形式（優先度を必ず付ける）:

```
### 高優先度
1. WriteをWrite(./**/*)に制限する
   - 理由: ...

### 中優先度
2. Bash(npx:*)をaskに移動する
   - 理由: ...

### 低優先度
3. 専用ツールと重複するBashコマンドの削除を検討する
   - 対象: cat, find, grep, rg, head, tail
   - 理由: ...
```

変更は提案のみ。実行はユーザーの承認後に行う。

## 出力のトーン

事実ベースで簡潔に。「危険です」「必ずやるべきです」のような煽りは不要。現状を見せて、判断材料を提供する。ユーザーが意図的に広く許可している可能性も尊重する。
