# 理解負債ワークフロー

理解負債 = 説明された直後は「わかった気がする」が、あとから自分で説明できない状態。
このワークフローは理解負債を「見つけて終わり」にせず、解消まで導くためのもの。

## 対話ルール（検知）

- 重要な処理・設計・データの流れを説明したら、直後に**確認質問を1つ**する
- ユーザーの回答が曖昧なら、どの概念かではなく**どこで詰まったか**を具体的に特定する
- AIが勝手に「理解した」と判定しない。**確認質問への回答だけ**を判定材料にする
- 検知した理解負債は `~/work/claude-obsidian/debt/<リポジトリ名>.md` に記録する（なければ作成。
  リポジトリ名は `git rev-parse --show-toplevel` のbasename。リポジトリ外なら `general.md`）

## 理解負債の記録形式（~/work/claude-obsidian/debt/<リポジトリ名>.md）

各項目は「小さなissue」として扱う。titleだけでなくclose条件まで書く:

```markdown
## Lv1 <テーマ名>
- 何がわかっていないか: <具体的に>
- 解消条件: <どの状態になったら解消とみなすか>
- 確認方法: <どう質問すれば確認できるか>
- 履歴:
  - 2026-07-03 #understanding-fail 入力値が途中で別形式に変換され次の処理へ渡される流れを説明できなかった
```

- レベルは3段階: **Lv1**（未理解）→ **Lv2**（説明できる）→ **Lv3**（コードを読める）
- 正答しても**1段階だけ昇格**。一気に完了にしない（「説明はできるがコードは追えない」中間状態を扱うため）
- 不正答は「間違い」ではなく**失敗ポイント**: `#understanding-fail` でどのデータ変換・どの分岐を追えなかったかを具体的に残す
- `#understanding-learned` の記録は慎重に。以下を説明できた場合のみ:
  - 内部ロジック / 入力から出力までのデータ変換 / 設計理由 / edge case・失敗条件
  - 「なんとなくわかった」は理解済みにしない
- Lv3に到達した項目は履歴ごと末尾の「解消済み」セクションへ移動

## commit前チェック

- commit・作業完了の前に理解負債ファイルにLv3未満の項目が残っていれば、
  「未解消の理解負債があります: <テーマ一覧>」と1行通知する
- **commitは止めない**。曖昧なまま閉じようとしていることに気づかせるだけ
- ユーザーが望めば /debt-check で1つだけ確認する
