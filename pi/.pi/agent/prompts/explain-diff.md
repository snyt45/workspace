# Explain Diff

指定されたコード変更について、リッチでインタラクティブな解説を作成してください。

以下のセクションで構成すること:

- **Background（背景）**: この変更に関連する既存システムを説明する。周辺コードを幅広く探索すること。読者が事前知識をどれくらい持っているかわからないため、初心者向けの深い背景（すでに知っている読者はスキップ可）と、変更に直接関連する絞り込んだ背景の両方を含める。
- **Intuition（直感）**: コード変更の核心となる直感を説明する。全詳細ではなく本質の伝達に集中する。具体例・おもちゃデータを使い、図やダイアグラムを積極的に使う。
- **Code（コード解説）**: 変更のハイレベルなウォークスルーを行う。理解しやすい順序・グループでまとめる。
- **Quiz（クイズ）**: PRの内容を理解できているか確認する5問のクイズを作る。適度な難易度（本質を理解しないと答えられないが、引っかけにしない）。インタラクティブな選択肢形式で、クリック時に正誤フィードバックを表示する。

フォーマット:

- CSSとJavaScriptを含む自己完結したHTMLファイルを1つ出力する。テーブルオブコンテンツ付きの1ページ構成。レスポンシブ対応（スマホでも見られる）。ファイルはコードリポジトリの外のグローバルな場所に置き、ファイル名は必ず `YYYY-MM-DD-` から始める（例: `/tmp/2026-01-12-explanation-<slug>.html`）。
- Martin Kleppmann のような明快さとフローで書き、読んでいて楽しい文章にする。セクション間の遷移はスムーズに。
- ダイアグラムのヒント:
  - 説明全体で再利用できる少数の「ダイアグラムファミリー」を選ぶと良い。
  - UIの変更ならアプリ画面の簡略版、データフロー・コンポーネント間通信ならシステム図（サンプルデータ付き）が有効。
  - ASCIIダイアグラムは使わない。HTMLデザインで表現する。
  - コードブロックは必ず `<pre>` タグを使うか、カスタムdivには `white-space: pre-wrap` を付ける。保存前に各コードブロックのCSSを確認すること。
- 重要概念・定義・エッジケースにはコールアウトを使う。
