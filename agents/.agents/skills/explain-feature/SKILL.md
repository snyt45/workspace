# Explain Feature

指定された機能について、コードベースを探索し、リッチでインタラクティブな解説を作成してください。explain-diffの姉妹スキルで、対象がdiffではなく「いま存在する機能そのもの」である点が違います。

## 調査フェーズ

HTMLを書き始める前に、機能の全体像を掴むこと:

1. **エントリポイントの特定**: 機能が起動するすべての入口を探す。HTTPルート（routes定義）、コントローラ、バックグラウンドジョブ、バッチ・cron、Webhook、CLIコマンド、フロントエンドのイベントハンドラなど。1つ見つけて満足せず、同じ機能に通じる入口を全部洗い出す。
2. **呼び出しチェーンの追跡**: 各エントリポイントから処理を辿り、関連ファイルを収集する。サービス・モデル・ビュー・シリアライザ・バリデーション・設定など、機能の理解に必要なファイルをレイヤーごとに整理する。
3. **データモデルの把握**: schema・マイグレーション・モデル定義から、機能が読み書きするテーブルとリレーションを特定する。
4. 機能が大きすぎる場合は、主要な処理経路（ハッピーパス）を軸に据え、周辺経路は簡潔に触れるに留める。

探索範囲が広いときは、並列サブエージェントでレイヤーごと・エントリポイントごとに分担すると速い。

## HTMLの構成

以下のセクションで構成すること:

- **Overview（概要）**: この機能がユーザー視点で何をするのか、なぜ存在するのかを説明する。読者の事前知識がわからないため、初心者向けの深い背景（すでに知っている読者はスキップ可）と、コードを読むうえで直接必要な絞り込んだ背景の両方を含める。
- **Entry Points（エントリポイント）**: 機能が起動するすべての入口を一覧にする。種別（HTTP / ジョブ / バッチ / Webhook / UIイベント等）、トリガー条件、`file:line` を明記する。
- **File Map（ファイルマップ）**: 関連ファイルをレイヤー別（エントリ / ビジネスロジック / データ / ビュー / 設定等）にグループ化した一覧。各ファイルに役割の1行説明を付ける。「この機能を触るとき最初に開くべきファイル」がわかるようにする。
- **Data Flow（データフロー）**: 入口 → 処理 → 永続化 → レスポンスの流れを図で表現する。具体的なサンプルデータ（おもちゃデータ）が各ステップでどう変換されるかを見せると理解が速い。分岐（エラー系・条件分岐）のうち重要なものも示す。
- **ER Diagram（ER図）**: 機能が読み書きするテーブルがあれば、主要カラムとリレーション（1:N、N:N等）をHTML/CSSまたはインラインSVGで図示する。全カラムは載せず、機能の理解に効くカラムに絞る。テーブルを使わない機能ならこのセクションは省略してよい。
- **Code Walkthrough（コード解説）**: 主要な処理経路のハイレベルなウォークスルー。理解しやすい順序・グループでまとめ、実際のコード片を引用しながら説明する。
- **Quiz（クイズ）**: 機能を理解できているか確認する5問のクイズ。適度な難易度（本質を理解しないと答えられないが、引っかけにしない）。インタラクティブな選択肢形式で、クリック時に正誤フィードバックを表示する。

## フォーマット

- CSSとJavaScriptを含む自己完結したHTMLファイルを1つ出力する。テーブルオブコンテンツ付きの1ページ構成。レスポンシブ対応（スマホでも見られる）。ファイルはコードリポジトリの外のグローバルな場所に置き、ファイル名は必ず `YYYY-MM-DD-` から始める（例: `/tmp/2026-01-12-feature-<slug>.html`）。
- Martin Kleppmann のような明快さとフローで書き、読んでいて楽しい文章にする。セクション間の遷移はスムーズに。
- ダイアグラムのヒント:
  - 説明全体で再利用できる少数の「ダイアグラムファミリー」を選ぶと良い。データフロー図とER図で色や形の意味を揃えると読者が迷わない。
  - ASCIIダイアグラムは使わない。HTMLデザインで表現する。
  - コードブロックは必ず `<pre>` タグを使うか、カスタムdivには `white-space: pre-wrap` を付ける。保存前に各コードブロックのCSSを確認すること。
- ファイルパスや `file:line` は読者がエディタで開けるよう正確に書く。
- 重要概念・定義・エッジケース・「ここがハマりどころ」にはコールアウトを使う。
