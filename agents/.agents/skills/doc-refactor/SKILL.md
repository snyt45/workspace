# Doc Refactor

コードをいきなり触らず、まずドキュメントに吐き出すことで責務の歪みを可視化し、ドキュメント上で合意した「あるべき姿」との差分を仕様としてリファクタリングする。流れが複雑なコードは、文章にした瞬間に「1つのモジュールの責務が一言で言えない」「同じ知識が2箇所にある」といった歪みが露呈する — それを利用するスキル。

3ステップで進める。軽量さが命なので、explain-featureのようなリッチHTMLは作らない。すべて短いMarkdownで、チャット出力または一時ファイル1枚に収める。

## Step 1: 現状マップを吐き出す

対象範囲のコードを読み、現状を画面1〜2枚のMarkdownにまとめる:

- **モジュールと責務**: 登場するファイル/クラス/関数それぞれに「一言の責務」を付ける。一言で言えない、「〜と〜と〜を担当」になる、名前と実態がズレている — これらが責務混在のサイン。無理に一言にせず、言えなかった事実をそのまま書く（それが発見）。
- **処理フロー**: 入口から出口までのステップ列。各ステップに担当モジュールを付記する。フローが行ったり来たりする（A→B→A→C）箇所は目立たせる。
- **におい一覧**: 上記から見えた問題を列挙する。責務の混在、レイヤーの飛び越し、双方向依存、同じ知識の重複、データの持ち回りすぎ、など。

調査は対象機能に絞り、深掘りしすぎない。ここで問題が見つからなければ「構造は健全、リファクタリング不要」と報告して終わってよい。

## Step 2: あるべき姿への変更案を出す

現状マップと同じフォーマットで「After」を書き、差分を明示する:

- Afterでは各モジュールの責務が一言で言えるように再配置する。
- **差分一覧**を必ず付ける: 「責務Xを A から B へ移動」「C と D を統合」「E を新設」「F は変更しない」の形式。変更しない部分を明記してスコープを絞る。
- 挙動（外から見た振る舞い）は変えない前提で設計する。機能変更が必要だと気づいた場合は別件として切り出す。
- 変更案が大きくなりすぎる場合は、独立して適用できる単位に分割し、今回やる範囲を提案する。

**ここでユーザーに確認を取る。ドキュメントの合意 = 仕様の合意であり、合意なしにStep 3へ進まない。** ユーザーが変更案を直したら、直された案が仕様になる。

## Step 3: 差分を仕様としてコードを修正する

- 合意したAfterマップを仕様として、差分一覧を1項目ずつ適用する。
- 各項目の適用後にテストを実行し、挙動が変わっていないことを確認する。テストがない場合は最初にその旨を伝え、最低限の安全網（既存挙動のスナップショット的な確認手段）を相談する。
- 完了後、Afterマップと実コードが一致しているかを見直し、差分一覧の各項目に完了/見送りを付けて報告する。見送りが出た場合は理由を書く。

## 原則

- **さっと使える軽さを維持する**: ドキュメントは作業のための足場であり、成果物ではない。装飾より速度。
- **ドキュメントが先、コードが後**: Step 1を飛ばして「すぐ直せそう」と思っても飛ばさない。文章化で見える歪みを取りこぼす。
- **1回のスコープを小さく**: 大規模な作り直しには使わない。分割して回数で刻む。
