# Explain

指定された対象（コード変更 or 既存機能）を分析し、リッチでインタラクティブな解説HTMLを作成する。

## モード判定

- PR・branch・diff・commitなど「変更」が対象 → **変更モード**
- 機能名・「〇〇の仕組み」など「いま存在するコード」が対象 → **機能モード**
- 両方の性質を持つ依頼（例:「このPRが触っている機能の全体像から知りたい」）は、機能モードの構成に変更モードのセクションを差し込むハイブリッドでよい

## 変更モード

以下のセクションで構成する:

- **Background（背景）**: この変更に関連する既存システムを説明する。周辺コードを幅広く探索すること。読者が事前知識をどれくらい持っているかわからないため、初心者向けの深い背景（すでに知っている読者はスキップ可）と、変更に直接関連する絞り込んだ背景の両方を含める。
- **Intuition（直感）**: コード変更の核心となる直感を説明する。全詳細ではなく本質の伝達に集中する。具体例・おもちゃデータを使い、図やダイアグラムを積極的に使う。
- **Code（コード解説）**: 変更のハイレベルなウォークスルーを行う。理解しやすい順序・グループでまとめる。
- **Quiz（クイズ）**: 共通セクション参照。

## 機能モード

### 調査フェーズ

HTMLを書き始める前に、機能の全体像を掴むこと:

1. **エントリポイントの特定**: 機能が起動するすべての入口を探す。HTTPルート（routes定義）、コントローラ、バックグラウンドジョブ、バッチ・cron、Webhook、CLIコマンド、フロントエンドのイベントハンドラなど。1つ見つけて満足せず、同じ機能に通じる入口を全部洗い出す。
2. **呼び出しチェーンの追跡**: 各エントリポイントから処理を辿り、関連ファイルを収集する。サービス・モデル・ビュー・シリアライザ・バリデーション・設定など、機能の理解に必要なファイルをレイヤーごとに整理する。
3. **データモデルの把握**: schema・マイグレーション・モデル定義から、機能が読み書きするテーブルとリレーションを特定する。
4. 機能が大きすぎる場合は、主要な処理経路（ハッピーパス）を軸に据え、周辺経路は簡潔に触れるに留める。

探索範囲が広いときは、並列サブエージェントでレイヤーごと・エントリポイントごとに分担すると速い。

### セクション構成

- **Overview（概要）**: この機能がユーザー視点で何をするのか、なぜ存在するのかを説明する。初心者向けの深い背景（スキップ可）と、コードを読むうえで直接必要な絞り込んだ背景の両方を含める。
- **Entry Points（エントリポイント）**: 機能が起動するすべての入口を一覧にする。種別（HTTP / ジョブ / バッチ / Webhook / UIイベント等）、トリガー条件、`file:line` を明記する。
- **File Map（ファイルマップ）**: 関連ファイルをレイヤー別（エントリ / ビジネスロジック / データ / ビュー / 設定等）にグループ化した一覧。各ファイルに役割の1行説明を付ける。「この機能を触るとき最初に開くべきファイル」がわかるようにする。
- **Data Flow（データフロー）**: 入口 → 処理 → 永続化 → レスポンスの流れを図で表現する。具体的なサンプルデータが各ステップでどう変換されるかを見せると理解が速い。重要な分岐（エラー系・条件分岐）も示す。
- **ER Diagram（ER図）**: 機能が読み書きするテーブルがあれば、主要カラムとリレーション（1:N、N:N等）を図示する。全カラムは載せず、機能の理解に効くカラムに絞る。テーブルを使わない機能なら省略してよい。
- **Code Walkthrough（コード解説）**: 主要な処理経路のハイレベルなウォークスルー。理解しやすい順序・グループでまとめ、実際のコード片を引用しながら説明する。
- **Quiz（クイズ）**: 共通セクション参照。

## 共通: クイズ

内容を理解できているか確認する5問のクイズを最後に置く。適度な難易度（本質を理解しないと答えられないが、引っかけにしない）。インタラクティブな選択肢形式で、クリック時に正誤フィードバックを表示する。

## 共通: フォーマット

- CSSとJavaScriptを含む自己完結したHTMLファイルを1つ出力する。テーブルオブコンテンツ付きの1ページ構成。レスポンシブ対応（スマホでも見られる）。
- 出力先は `~/explains/YYYY-MM-DD-<slug>.html`（ディレクトリがなければ作成する）。
- Martin Kleppmann のような明快さとフローで書き、読んでいて楽しい文章にする。セクション間の遷移はスムーズに。
- ダイアグラムのヒント:
  - 説明全体で再利用できる少数の「ダイアグラムファミリー」を選ぶと良い。複数の図で色や形の意味を揃えると読者が迷わない。
  - UIの変更ならアプリ画面の簡略版、データフロー・コンポーネント間通信ならシステム図（サンプルデータ付き）が有効。
  - ASCIIダイアグラムは使わない。HTMLデザインで表現する。
  - コードブロックは必ず `<pre>` タグを使うか、カスタムdivには `white-space: pre-wrap` を付ける。保存前に各コードブロックのCSSを確認すること。
- ファイルパスや `file:line` は読者がエディタで開けるよう正確に書く。
- 重要概念・定義・エッジケース・「ここがハマりどころ」にはコールアウトを使う。

## 出力後のワークフロー

1. **lavishで開く**: `lavish-axi <file>` で解説HTMLを開く（コマンドがなければ `open` にフォールバックし、その旨を伝える）。live reload対応なので、指摘を受けたらHTMLファイルを直接修正すれば自動反映される。アノテーションの回収は `lavish-axi poll`（詳細はlavishスキル参照）。
2. **Obsidianへの保存（求められたら）**: 「Obsidianに保存して」と言われたら、解説のMarkdown版を作成し、obsidian-vaultスキル/MCPでvaultの `explains/` フォルダに保存する（ファイル名はHTMLと同じ `YYYY-MM-DD-<slug>.md`）。HTMLを機械的に変換するのではなく、Markdownとして自然な形（見出し・コードブロック・Mermaid図）に書き直す。インタラクティブなクイズは動かないので、質問と解答を折りたたみ（`> [!question]` コールアウト等）で表現するか省略する。
